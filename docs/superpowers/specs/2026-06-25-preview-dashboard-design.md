# Дизайн: превью-харнесс дашборда по фазам

_дата: 2026-06-25 · статус: утверждён дизайн, готов к плану_

## Проблема

Нужно в любой момент видеть **актуальный** вид дашборда, который генерят агенты,
во всех фазах, где он выглядит по-разному — без запуска настоящего `/feature`.
Сейчас единственный способ увидеть дашборд — дождаться реального прогона.

## Решение (одной фразой)

Команда `python dev.py preview` наполняет текущий `templates/dashboard.html`
готовыми демо-данными для каждой фазы, поднимает companion-сервер и открывает хаб
со списком демо-задач.

## Архитектура

### 1. Где живут фикстуры (в гите)

`.workflow/` **в `.gitignore`**, поэтому durable-данные фикстур кладём в
коммитимый каталог: **`templates/fixtures/<slug>/`**.

Четыре фикстуры — по одной на фазу с различимым видом:

| slug | фаза / состояние | что показывает |
|---|---|---|
| `_preview-elaborate` | ELABORATE · awaiting | план-блоки, вопросы (choice + свой ответ), actionbar в `awaiting`, анкорный тред чата с `needsAnswer`, чип work-stream'ов, левый сайдбар |
| `_preview-implement` | IMPLEMENT · working | work-stream'ы (микс done/working), subagents, прогресс-бар, строка «Сейчас: …», живая лента трейсинга |
| `_preview-done` | DONE | итог-сводка, коммит + verification, план-блоки ✅, diff на «Изменениях» |
| `_preview-ask` | ANSWER (`/ask`) | read-only сводка, `demo` + `mockups` (SVG-инфографика + процесс-диаграмма в sandbox-iframe), бейдж `kind:"ask"` в хабе, чат |

Каждый каталог фикстуры содержит файлы, которые отдаёт сервер:
`state.json`, `dashboard.json`, `replies.json`, `signals.json`, `chat.jsonl`,
`telemetry.jsonl`, плюс `mockups/` для ask-фикстуры.
**`index.html` НЕ хранится** — он генерится (см. п.2).

### 2. Харнесс — `scripts/preview.py` (только stdlib, по конвенциям проекта)

Шаг `install`, идемпотентный:

1. Скопировать каждый `templates/fixtures/*` → `.workflow/tasks/*`.
2. Скопировать **текущий** `templates/dashboard.html` → каждый `.../index.html`
   — так превью всегда отражает живой шаблон.
3. Закрепить `state.baseCommit` фиксированным старым коммитом в implement/done
   фикстурах, чтобы вкладка **«Изменения»** показывала реальный непустой diff.

Затем (пере)запустить companion-сервер (через существующую singleton-логику
`scripts/server.py`) и открыть **`/hub`** — оттуда кликом заходишь в любую
фазовую фикстуру.

### 3. Одна команда — `python dev.py preview`

Тонкий сабкоманд в `dev.py`, оборачивающий харнесс:
`python dev.py preview` → обновить фикстуры + (пере)поднять сервер + открыть хаб.
Перезапуск после правок шаблона показывает новый вид.

Флаги (минимум): `--no-browser` (не открывать браузер), `--port N` (как у `serve`).

### 4. Тесты

Один smoke-тест в `tests/`: для каждой фикстуры — JSON валиден и есть
обязательные ключи (`slug`, `phase`, `title` в `state.json`/`dashboard.json`).
Дешёвый гард: кривая фикстура падает в CI, а не рендерится пустой страницей.

## Контракт данных (откуда берутся поля)

Источник истины — `scripts/server.py` (эндпоинты `/state.json`, `/data`→`dashboard.json`,
`/changes`, `/trace/feed`, `/chat.jsonl`, `/mockup`) и реальные задачи в
`.workflow/tasks/*`. Фикстуры копируют форму этих файлов с демо-содержимым.

## Известные ограничения / нюансы

- **Вкладка «Изменения»** — единственная не полностью контролируемая поверхность:
  строит diff из живого git (`git diff baseCommit` в корне задачи). Закреплённый
  `baseCommit` делает её детерминированной-ish, но машинно-зависимой (зависит от
  истории репо). Остальные вкладки — полностью на демо-данных. Это документируется.
- **Общий рабочий дерево.** Этот репо — исходник плагина, конкурентные сессии
  делят одно рабочее дерево. Добавление коммитимых фикстур + скрипта —
  низко-коллизионное, отдельный worktree не нужен.
- **Имена `_preview-*`** появятся в хабе рядом с реальными задачами — это намеренно
  (видно и хаб, и карточки). Префикс `_preview-` делает их очевидно демо-шными.

## Вне охвата (YAGNI)

- Скриншот-харнесс / визуальная регрессия (отдельная задача при необходимости).
- Автогенерация фикстур из реальных прогонов — фикстуры пишутся руками один раз.
- Покрытие промежуточных под-состояний (EXPLORE, VERIFY) — 4 фазы дают
  достаточный визуальный охват.
