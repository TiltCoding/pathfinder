# Дизайн: видимость процесса + диалог в контексте на дашборде задачи

> Дата: 2026-06-19 · Статус: одобрен к планированию
> Скоуп: первый срез улучшений дашборда — **A1 + A2 + B1**.

## Контекст и цель

Дашборд задачи (`templates/dashboard.html` + `scripts/server.py`) — это
**почтовый ящик + блокирующее ожидание**, а не пульт. Сервер (stdlib HTTP)
только читает/пишет файлы; активная сторона — сессия Claude Code в терминале,
которая на чекпоинтах опрашивает состояние и забирает почту (`/wait`).

Пользователь, наблюдая за работой агента, назвал три боли:
1. **Видимость** — непонятно, что агент делает прямо сейчас и какой прогресс по работам.
2. **Фидбек** — обсуждение односторонне: нельзя задать встречный вопрос по
   конкретному месту и получить ответ в том же контексте.
3. Действия в UI (запуск/очередь задач, редактирование артефактов) — **вынесено
   в отдельные циклы**, не входит в этот срез.

Этот срез закрывает боли 1 и 2 тремя улучшениями: **A1** («Сейчас: …»),
**A2** (сводка работ), **B1** (обсуждение на якоре).

## Не-цели (явно вне скоупа)

- C1 (inline-редактирование артефактов), C2 (очередь задач из хаба) — отдельные циклы.
- Управление ходом агента (пауза/стоп/прерывание) — архитектурно невозможно
  без демона, отвергнуто.
- Карта всех фаз «вы здесь» — пользователь подтвердил, что не нужна.
- Ужесточение правил заполнения агента как самоцель — не та боль; правила
  добавляются лишь как лёгкий механизм для A1.

## Архитектурный принцип

Всё едет **поверх существующего контракта**: агент заполняет поля/строки,
дашборд их рендерит и перерисовывает по изменению `lastDataStamp`. Правка
`server.py` ровно одна (B1: прокинуть `anchor`/`needsAnswer` в `_chat_post`).
Соответствует ADR-0008/0013/0016 (бэкенд агностичен к содержимому).

## A1 — строка «Сейчас: …»

**Контракт.** Агент пишет в `dashboard.json`:
- `now` — человеческая фраза текущей активности («исследую `server.py`»,
  «пишу тесты для `/trace`»).
- `nowAt` — ISO-метка момента обновления `now`.

**UI.** В шапке, под `#status`, строка вида `Сейчас: исследую server.py · 12с назад`.
В состоянии `awaiting-batch` строка заменяется существующим «ждёт ваши правки».

**Edge-cases.**
- Свежесть: если `nowAt` старше ~90 с и статус `working` — приглушаем до
  `Сейчас: …` (не врать устаревшим).
- Стамп: добавить `data.now` в ключ `lastDataStamp` (стр. ~2356), иначе строка
  не перерисуется.
- Отсутствие `now`: строку скрываем (старые задачи без поля не ломаются).

**Заполнение агентом.** Лёгкое правило в скилл-оркестраторы (`/feature`,
`/new-product`, `/improve`, `/ask`): обновлять `now` на смене активности. Это
единственное новое «правило заполнения» в срезе — намеренно лёгкое.

**Затронуто.** `templates/dashboard.html` (рендер шапки, стамп); тексты
оркестраторов (`skills/*/SKILL.md` или reference-файлы) — добавить пункт про `now`.

## A2 — сводка работ

**Контракт.** Данные уже есть: `workstreams[{status, title}]`
(`status` ∈ `todo|in_progress|done`). Агент ничего нового не пишет.

**UI.** Компактный чип рядом с прогресс-баром в шапке:
`▓▓▓░░ 3/7 готово · 1 в работе · 3 в очереди`. Цвета — существующие токены
(`--ok` / `--accent` / `--muted`). Карточка «Work-streams» остаётся как
детализация; чип — её свёртка наверху.

**Источник правды прогресс-бара.** Сейчас `#progress-bar` движется от
`data.progress {done,total}`. Решение:
- если `workstreams` непусто → прогресс выводим из них (`done / total работ`);
- иначе → как сейчас (`data.progress`).

Это убирает возможное противоречие между баром и чипом.

**Затронуто.** `templates/dashboard.html` (рендер шапки + функция расчёта
прогресса).

## B1 — обсуждение на якоре (threaded discussion)

### Бэкбон — anchored chat

Расширяем `chat.jsonl`: у сообщения появляется опциональное поле `anchor`
(значение = `blockId` плана / `vr.id` демо-варианта / `summary` / `codebaseMap`).
- Сообщения **с** `anchor` рендерятся веткой под своим блоком/регионом.
- Сообщения **без** `anchor` — в глобальной чат-панели (как сейчас).

Агент отвечает в тот же `chat.jsonl` с тем же `anchor`. Обе стороны диалога
персистятся по построению — снимается проблема «реплика человека исчезает
после submit».

### Ключевой инсайт: batch vs immediate растворяется

Хотя `POST /chat` отправляет сообщение сразу, **агент читает только на
чекпоинте**. Значит человек пишет по ходу, а агент потребляет накопленное
пачкой на ближайшем чекпоинте — получаем и немедленную отправку, и
батч-семантику без выбора между ними. Глобальный `Submit/Approve` остаётся
исключительно для утверждения плана; обсуждение — параллельный канал,
доступный и в `working`, и в `awaiting`.

### Слияние select-to-comment в ветки (решение пользователя)

Нынешний путь «выделить текст → коммент» (draft `kind` с `blockId` +
`selectedText` → batch submit → агентская реплика) **сливается** в anchored-ветки:
- выделение текста становится **первым сообщением** anchored-ветки;
  `selectedText` переносится как цитата-метаданные первого сообщения;
- сообщение постится **сразу** через `POST /chat` (`anchor` = `blockId`).

Следствия:
- `draftItems` остаётся только для `kind:"answer"` (ответы на вопросы) и выбора
  демо-варианта; черновики-комментарии исчезают.
- Счётчик «Очередь правок» (`updateQueue`) перестаёт считать комментарии.
- `#sel-popover`, `cmtCard`, draft-ветка в `regionFooter` перенаправляются на
  `POST /chat` с `anchor`; функции draft-комментариев удаляются.
- Коммент к демо-варианту (`v-comment` textarea + `saveVariantComment`) — та же
  механика; тоже становится anchored-сообщением (`anchor = vr.id`) и сливается
  в ветку под вариантом, а не отдельный путь.

### Встречный вопрос агента

Агентская реплика с флагом `needsAnswer: true` — это вопрос, пришпиленный к месту.
Ветка считается **открытой**, пока в ней нет более позднего (`ts`) сообщения
`role:"human"` после такого вопроса.
- На блоке с открытой веткой — жёлтая рамка-маркер.
- В шапке (рядом со сводкой A2) — счётчик `🔸 N ждут вашего ответа`.
- Ответ человека в ветке закрывает её.

### UI ветки (под блоком, вместо плоского `regionFooter`)

- Турны в хронологии по `ts`; визуально различаем «вы» / «агент».
- Цитата выделенного фрагмента — над первым сообщением (если был `selectedText`).
- Инлайн-поле «ответить в этой ветке» в конце — постит anchored-сообщение сразу.
- Статус-строка «учтено агентом», когда после реплики человека пришёл
  агентский ответ в той же ветке.

### Серверная правка (единственная во всём срезе)

`_chat_post` (`scripts/server.py`, ~стр. 1148) прокидывает в `msg` поля
`anchor` и `needsAnswer` из `body` (агностично, ~2 строки). `GET /chat` уже
отдаёт сообщения целиком — менять не нужно.

**Замечание о записи агентом.** Агент пишет свои турны (включая `needsAnswer`,
`anchor`) напрямую в `chat.jsonl` через свой инструментарий — как делает сейчас.
Правило фиксируется в оркестраторах.

### Затронуто

- `scripts/server.py` — `_chat_post` (+ `anchor`, `needsAnswer`).
- `templates/dashboard.html` — рендер веток под регионами/блоками/вариантами;
  перенаправление popover/select-to-comment на anchored chat; маркер открытой
  ветки; счётчик в шапке; удаление draft-комментариев и их учёта в очереди;
  чтение `/chat` в цикле поллинга (сейчас грузится только по открытию панели —
  нужно подтягивать для веток постоянно) + включение его в стамп.
- Тексты оркестраторов — правило про `anchor`/`needsAnswer`.

## Тестирование

В духе `conventions.md` (оффлайн `unittest`, stdlib-only, кросс-платформенно):
- `tests/test_chat_anchor` — `POST /chat` с `anchor`/`needsAnswer` сохраняет поля
  в `chat.jsonl`; `GET /chat` их возвращает; пустой `text` по-прежнему 400.
- Сигнал `chat` и пробуждение `/wait` не сломаны прокидыванием новых полей.
- A1/A2 — клиентские (рендер/стамп); проверяются вручную через `/run` +
  фикстурный `dashboard.json` с `now`/`nowAt` и `workstreams`. Серверных
  изменений нет — юнит-тестов сервера не требуют.

## Риски и заметки

- **Поллинг `/chat`.** Сейчас чат подтягивается лениво (по открытию панели).
  Для веток его нужно тянуть в общем цикле — следить за нагрузкой и стампом
  (не перерисовывать без изменений). Кэш на сервере для `/chat` отсутствует —
  файл маленький, чтение дёшево, но учесть при росте.
- **Дисциплина агента.** A1 (`now`) и встречные вопросы (`needsAnswer`) зависят
  от того, что агент их пишет. Лёгкие правила в оркестраторах; деградация
  мягкая (нет поля — нет строки/маркера, UI не ломается).
- **Совместимость со старыми задачами.** Все новые поля опциональны; задачи без
  них рендерятся как раньше.
