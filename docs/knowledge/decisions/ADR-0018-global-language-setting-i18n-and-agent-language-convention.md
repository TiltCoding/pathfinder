# ADR-0018: Глобальная языковая настройка + i18n дашборда/хаба + конвенция языка агента

- **Дата:** 2026-06-19
- **Статус:** принято
- **Задача:** dashboard-i18n

## Контекст

UI дашборда (`templates/dashboard.html`) и хаба (`HUB_PAGE`/`INDEX_LANDING` в `scripts/server.py`) был
**целиком захардкожен по-русски** (~244 строки кириллицы в JS-шаблонах рендера дашборда + ~30 в хабе),
i18n-слоя не было. Конвенция агента жёстко требовала «всё человекочитаемое → русский». Задача из трёх
частей (решения INTAKE от человека):

1. **eng-first UI** — строки дашборда и хаба по умолчанию английские, русский — переключаемый язык
   (пока только `en`/`ru`).
2. **Глобальная языковая настройка плагина**, управляемая **из хаба**: один язык на весь плагин (не
   per-project), персистится, применяется к UI и задаёт дефолтный язык артефактов агента.
3. **Язык ответа агента** — в чате/реплаях агент отвечает на языке вопроса (авто-детект перебивает
   дефолт), машинные заголовки/ключи остаются английскими.

Переключатель темы (ADR-0015) был точным шаблоном механики (localStorage-ключ, общий хаб↔задача,
head-bootstrap, иконка-кнопка в шапке, дублирование логики в двух файлах), но тема меняет только
CSS-переменные, а язык требует подмены ~270 текстов через словарь + `t(key)` внутри рендер-функций.

## Решения

### 1. Глобальный конфиг в `~/.claude/ai-pathfinder/settings.json`
- Формат `{"lang": "en"|"ru", "ts": <iso>}`, дефолт `en`, белый список `{en, ru}`, деградация к `en` при
  любой ошибке/неизвестном значении.
- Хелперы `settings_path()` / `read_lang()` / `write_lang(lang)` в `scripts/server.py` (с
  override-параметром базы для тестовой изоляции — НЕ писать в реальный `~/.claude/`).
- **Почему `~/.claude/`, а не per-project/`.workflow` и не плагин-рут:** настройка **глобальная**
  (один язык на весь плагин). `.workflow` — per-project store (общий через симлинк, ADR-0010), туда
  глобальная настройка не ложится. **Плагин-рут не годится принципиально — это кэш-копия** (репо =
  исходник, `~/.claude/plugins/cache/.../` — развёрнутая побайтово-идентичная копия, перетирается при
  refresh; ADR-0017). `~/.claude/` — уже принятая конвенция домашней папки (`_aipf.py:274`
  `expanduser("~/.claude/projects")`).

### 2. Единый канал `/settings.json` без кэша (vs дозапись в `/hub.json`)
- `GET /settings.json` — slug-free ветка в `do_GET` (рядом с `/queue.json`), без кэша (`_send` уже
  no-store), тело `{"lang": <en|ru>}`. **Единственный источник истины** для дашборда И хаба.
- **Почему отдельный эндпоинт, а не поле в `/hub.json`:** `/hub.json` имеет TTL-кэш `_hub` — язык,
  подмешанный туда, отдавался бы устаревшим, и хаб с дашбордом видели бы рассинхрон. Отдельный
  no-store эндпоинт гарантирует, что обе страницы читают одно живое значение → «меняется из hub»
  работает кросс-страница мгновенно.

### 3. `POST /settings` ДО проверки slug
- `POST /settings` — новая ветка в `do_POST`, валидирует `body.get("lang")` белым списком (иначе 400),
  пишет atomic, возвращает `{"ok": true, "lang": lang}`. Хаб — «писатель» глобальной настройки.
- **Почему ДО проверки slug:** `do_POST` отвергает запрос без slug (400). У `/settings` slug нет, поэтому
  ветку вставили **сразу после чтения body, до проверки slug** — иначе валидный POST падал бы 400.
  Зафиксировано как риск R1 в плане; покрыто тестом «POST без slug не падает».

### 4. eng-first дефолт
- При отсутствии настройки/значения — жёсткий `en` (не `navigator.language` авто-детект первого захода).
- **Почему:** бриф требует english-first как осознанный дефолт продукта; авто-детект по локали браузера
  дал бы непредсказуемый первый заход. Дефолт един в head-bootstrap и в `resolveLang()`.

### 5. Словарь + `t()` по образцу темы, дубль в двух файлах
- Инлайн-словарь `STR = { en:{...}, ru:{...} }` с **плоскими английскими ключами** (`t("tab.changes")`)
  + функция `t(key)`. Логика `storedLang()`/`resolveLang()`/`applyLang()`/`initLang()` — по образцу темы
  (ADR-0015). Источник истины — `/settings.json` (fetch при старте) + `localStorage['lang']` как кэш +
  поллинг для кросс-страничного синка. Смена языка триггерит перерисовку (re-`render`), т.к. динамические
  строки берутся из словаря на каждом тике поллинга.
- Контрол — **флип-иконка-кнопка `#lang-btn` (EN⇄RU)** рядом с `#theme-btn` (точный клон механики темы,
  выбран вместо segmented `.seg`).
- **Дубль словаря в `dashboard.html` и `HUB_PAGE`** (raw-строка, функции не шарятся — отдельные стримы,
  как у темы) → **риск дрейфа** наборов ключей. Закрыт **smoke-тестом полноты словаря** (оба языка =
  одинаковый набор ключей), по аналогии с инвариантом полноты тёмной палитры (ADR-0015,
  `--warn-soft`/`--awaiting-soft`).

### 6. Классификация текстов (часть 3)
- **Язык вопроса перебивает дефолт** для: `replies.json`, `chat.jsonl` (role:agent, вкл. anchored),
  `/ask` summary/planBlocks (весь артефакт = ответ).
- **Глобальный дефолт (eng-first)** для: brief/exploration/plan/questions, summary прочих скиллов,
  planBlocks, docs/knowledge, тексты гейтов, опции choice, коммит-сообщения, PRD/README продукта.
- **Стабильно английские (НЕ локализуются):** машинно-парсимые заголовки/ключи — секции дайджестов
  `ask-researcher`, `cand:`-ключи `wf-improver`, фиксированные схемы. **Почему:** их парсит оркестратор;
  перевод заголовка сломал бы парсер (то же обоснование, что у ADR-0012 — «свободные ответы ломают
  агрегацию»). Тела инструкций агентов тоже остаются английскими — меняется только конвенция
  human-facing **вывода**.
- **Контент агента из `dashboard.json` (title/summary/questions/now) НЕ локализуется фронтом** — это
  контент, а не хром UI. Язык контента задаёт сам агент по правилу выше.

### 7. Чтение настройки агентом + применение к рантайму
- INTAKE-шаг в каждом скилле (рядом со стартом сервера): прочитать `settings.json` (грейсфул → `en`),
  записать резолвленный язык в `state.json` (новое поле), передавать под-агентам в промпте спавна —
  оркестратор медиирует хэндофф (как slug/путь; субагенты не спавнят субагентов, ADR-0006).
- **Применение части 3 к рантайму — только после refresh плагина** (правки `skills/**`/`agents/**` живут
  в репо-исходнике, рантайм крутит кэш-копию; ADR-0017). В репо проверяется лишь корректность правок.

## Последствия

- Появился первый **глобальный** (вне `.workflow`/репо) персистентный конфиг плагина — будущие глобальные
  настройки складывать туда же (`~/.claude/ai-pathfinder/`).
- Конвенция языка в базе знаний и инструкциях изменена: «всё человекочитаемое → русский» заменено на
  «язык = глобальная настройка (`en` при отсутствии); чат/реплаи = язык вопроса». База знаний остаётся
  на русском **по факту настройки**, а не по жёсткому правилу.
- Словарь дублирован в двух файлах — правки синхронны, рассинхрон ловит smoke-тест. Расширение языков
  (>2) потребует только пополнения `STR` + smoke-теста, но контрол-флип рассчитан на 2 языка (для N
  понадобится segmented/dropdown).
- Несколько серверов (по одному на проект) пишут один глобальный `settings.json` — last-writer-wins,
  `os.replace` атомарен, гонка редкая (ручной клик), межпроцессный лок не нужен.

## Связи

- **ADR-0015** (тема) — точный образец механики: localStorage-ключ, head-bootstrap, иконка-кнопка,
  дубль логики в двух файлах, инвариант полноты (палитра ↔ набор ключей словаря).
- **ADR-0017** (singleton/heartbeat) — обоснование «плагин-рут = кэш-копия, правки видны после refresh».
- **ADR-0010** (общий store через симлинк) — почему глобальная настройка НЕ в `.workflow`.
- **ADR-0016** (`/ask`) — `/ask` summary/planBlocks классифицированы как «язык вопроса».
- **ADR-0012** (детерминированная агрегация) — то же обоснование «машинные заголовки стабильны»
  (свободный текст ломает парсер оркестратора).
- **ADR-0008/0013** (агностичный бэкенд) — фронт-i18n не трогает контракт `dashboard.json`.
