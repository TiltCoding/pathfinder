# ADR-0023: `/design` — фокусный UI/UX-аудит одного компонента → ОДИН аннотированный демо

- **Дата:** 2026-06-25
- **Статус:** принято
- **Задача:** design-command

## Контекст

Нужна пятая slash-команда плагина — **`/design <компонент>`**: глубокий **UI/UX-аудит ОДНОГО**
выбранного компонента интерфейса с последующей реализацией одобренных правок. Человек указывает **один**
элемент (по имени и/или скриншоту), команда критикует его сквозь **набор UI/UX-призм**, сводит находки в
**предложение правок**, после согласия человека — реализует только одобренные. Это **сиблинг**
`/feature`/`/new-product`/`/improve`/`/ask`, отличающийся тем, что **производит**: фокусный
аудит-и-редизайн **одного** компонента (не бэклог по всему приложению — для этого `/improve`; не read-only
объяснение — для этого `/ask`; не произвольная фича по спеке — для этого `/feature`).

Машина стадий: `INTAKE (имя и/или скриншот) → AUDIT (рой ds-auditor по 7 UI/UX-призмам) → COMPOSE
(оркестратор сводит находки в ОДИН аннотированный мокап) → CONSENT GATE (чек-лист по находкам,
feat-K-стиль, дефолт «Применить» + approve-plan) → IMPLEMENT (ds-coder применяет одобренные находки) →
VERIFY (wf-reviewer + /code-review) → DONE`.

Как и у `/ask`/`/improve`, разведка подтвердила: **ни один серверный эндпоинт не завязан на семантику
доменных полей** (ADR-0008/0013/0016) — модель дашборда чисто клиентская, сервер отдаёт JSON-байты.
Значит весь `/design` ложится на существующие контракты **с 0 правок `server.py`**.

## Решение

Строить `/design` целиком **поверх существующего контракта** (ride-the-contract, как ADR-0008/0013/0016)
с **0 правок сервера/HTML**, со следующими ключевыми развилками.

### 1. Единый аннотированный демо вместо N демо-вариантов

В COMPOSE оркестратор строит **один** самодостаточный `mockups/redesign.html`, покрывающий **все** находки
в стиле **«Вариант А»**: нумерованные бейджи ①②③, наложенные на редизайненный компонент + боковая легенда
(номер → проблема → что изменилось) + тогл **«До/После»**. Демо самодостаточно под CSP (инлайн-CSS/JS, без
CDN/сети; `data:`-картинки можно). `dashboard.json.demo` — **единственный** вариант
(`selectionId:"design-demo"`, `variants:[{id:"redesign",file:"redesign.html",…}]`), а **не** pick-one набор:
человек не выбирает «один из вариантов решения», а смотрит цельную картину предложенных правок и на гейте
отмечает, какие из них **применить**.

### 2. Гейт согласия — per-finding `f<k>`, дефолт opt-out «Применить»

CONSENT GATE — **не** «утвердить план», а **чек-лист по находкам** (feat-K-стиль ADR-0013). Каждая
консолидированная находка = одна карточка `planBlocks[]` + один `questions[kind:"choice"]` с тем же
`id = f<k>`, `options:["Применить","Пропустить"]`. **Дефолт — «Применить»** (opt-out): нет ответа =
находка применяется. Человек снимает галки с ненужных, **«Отправить»** (Submit) → **«Утвердить план»**
(Approve = «реализуй оставшийся набор»); обязательный порядок Submit → Approve. **0 правок сервера/HTML** —
реюз контракта `questions[choice]` + `approve-plan`.

### 3. Новый ростер `ds-*` без пина модели (два файла)

Заводим **отдельный** ростер: `ds-auditor` (read-only, `tools: Read, Grep, Glob, Bash` — без Write/Edit) и
`ds-coder` (`tools: Read, Write, Edit, Bash, Grep, Glob`). **Обе роли — без `model:`**: наследуют модель
сессии (человек выбрал «без пина» на интейке, как `wf-*`/`ask-researcher`). Два **отдельных** файла — из-за
**разного tool-set** (auditor строго read-only, coder пишет), а не из-за модели.

### 4. Вход компонента — имя и/или скриншот

Компонент задаётся **именем и/или скриншотом**. Имя → оркестратор находит компонент в коде (Grep/Glob).
Скриншот → человек прикрепляет изображение на дашборде (контракт вложений ADR-0020), оркестратор `Read`'ит
сохранённый файл и передаёт визуальный контекст аудиторам. Только-имя → работа от кода; только-скриншот →
от визуала + поиск парного кода; оба → перекрёстная сверка.

### 5. VERIFY гоняет только `/code-review`

В отличие от `/feature` (оба ревью-гейта), `/design` запускает **только** `/code-review` (плюс
`wf-reviewer`, реюз из feature). `/security-review` не гоняется — фокусные UI/UX-правки компонента редко
несут поверхность безопасности; при необходимости человек запустит гейт с дашборда вручную.

## Почему

- **Культ «0 правок» (ADR-0008/0013/0016).** Бэкенд агностичен к содержимому items → находки/демо/гейт
  ложатся на существующие поля (`planBlocks`/`questions[choice]`/`demo`/`approve-plan`) без новых
  эндпоинтов/флагов; риск рассинхрона фронт↔сервер↔агент нулевой. `/design` — **пятый** потребитель
  ride-the-contract (после `/feature`-фидбэка, `/new-product`-гейтов, `/improve`-гейта, `/ask`).
- **Один аннотированный демо честнее N демо.** Редизайн **одного** компонента — это **набор точечных
  правок**, а не альтернативные дизайны на выбор; показывать N конкурирующих вариантов было бы ложной
  моделью. «Вариант А» (бейджи + легенда + До/После) даёт человеку цельную картину «что и почему меняется»
  и прямо ложится на per-finding гейт: бейдж ①②③ в демо = карточка `f<k>` на гейте.
- **Opt-out дефолт «Применить» — намеренная инверсия `/improve`.** В `/improve` дефолт **opt-in** («нет
  ответа = Пропускаем»): из широкого бэклога человек осознанно выбирает немногое. В `/design` человек уже
  **сам указал** компонент и запросил его улучшение → разумный дефолт «применить всё, сними лишнее» (opt-out)
  снижает трение. Та же машина гейта (`feat-K`/`f<k>`), обратный дефолт.
- **Ростер `ds-*` без пина — следствие инварианта «model глобальна для subagent_type» (ADR-0006).** Реюз
  `wf-explorer`/`wf-coder` невозможен: их `description`/Output заточены под EXPLORE/IMPLEMENT
  `/feature`-задачи, а `/design` нужны UI/UX-призменные находки `{ id, prism, severity, problem, location,
  proposal }` и применение **уже решённой** находки. Своя пара даёт чистые `description`/контракты. Два
  файла — из-за **несовместимого tool-set** (read-only auditor vs Write coder), не из-за модели; пиннить
  модель не нужно — дефолта сессии хватает.
- **Скриншот через вложения дашборда (ADR-0020).** Визуальный вход компонента уже покрыт контрактом
  base64-`/attach` → ref → `GET /image`; оркестратор `Read`'ит сохранённый файл по абсолютному пути.
  Отдельного канала загрузки не требуется.

## Последствия

- **`/design` — лёгкий оркестратор, без `wf-documenter`.** Скоуп узкий (один компонент) → база знаний
  растится **лайтово** самим оркестратором (строка в `task-log.md`); полный проход документера не
  запускается.
- **`ds-auditor` read-only по конструкции.** Без Write/Edit — аудиторы не правят код и **не рисуют демо**;
  весь синтез (консолидация находок + единый аннотированный мокап) держит оркестратор (инвариант ADR-0006
  «субагенты не спавнят субагентов»).
- **Демо самодостаточно под CSP.** `redesign.html` — инлайн-CSS/JS, без CDN/сети; подаётся штатным
  `demo`-механизмом (`GET /mockup` в sandbox-iframe). Имя под маску
  `MOCKUP_RE = ^[A-Za-z0-9._-]{1,64}\.(html|svg)$`.
- **Бамп витрины.** `plugin.json` `0.22.0 → 0.23.0` (косметика — новые ассеты подключаются по конвенции
  каталогов, не из манифеста; `marketplace.json` не трогаем).
- **Коллизия нумерации ADR.** Задача изначально взяла `0022`, но при интеграции в `main` номер уже занял
  `ADR-0022-request-language-wins` (фича i18n, влитая параллельно), поэтому этот ADR перенумерован в
  **0023** (а версия плагина — в `0.23.0`, т.к. `0.22.0` выпущен под request-language).

## Альтернативы (отклонено)

- **N конкурирующих демо-вариантов (pick-one, как у `/feature`/`/ask`)** — отклонено: редизайн одного
  компонента — это набор точечных правок, а не альтернативные дизайны; pick-one — ложная модель. Единый
  аннотированный демо честнее и прямо ложится на per-finding гейт.
- **Альтернативные стили аннотации (выноски/оверлей-диффы вместо «Варианта А»)** — отклонено в пользу
  нумерованных бейджей ①②③ + легенда + тогл «До/После»: бейдж демо ↔ карточка `f<k>` гейта дают однозначную
  привязку «что изменится ↔ что одобряю».
- **Реюз `wf-explorer`/`wf-coder` вместо ростера `ds-*`** — отклонено: их контракты заточены под
  EXPLORE/IMPLEMENT `/feature`, не под UI/UX-призменный аудит; плюс модель глобальна для `subagent_type`
  (ADR-0006) — правка `wf-*` поменяла бы поведение `/feature`. Нужен свой ростер.
- **Один файл `ds-*` на обе роли (как `wf-improver` scout/vote)** — отклонено: auditor и coder требуют
  **несовместимый tool-set** (read-only vs Write/Edit), а один файл = один tool-set. Один файл оправдан
  только при общем tool-set и общей модели.
- **Пиннить модель `ds-*` (как `np-*`)** — отклонено: дефолта сессии хватает, лишний ростер с `model:` не
  нужен (ADR-0006).
- **Opt-in дефолт «Пропустить» (как `/improve`)** — отклонено: человек уже сам указал компонент и попросил
  улучшить → opt-out «Применить» снижает трение; инверсия осознанна.
- **Гонять `/security-review` в VERIFY** — отклонено: фокусные UI/UX-правки одного компонента редко несут
  поверхность безопасности; при нужде человек запустит гейт вручную с дашборда. Оставлен только
  `/code-review`.

## Связи

- **ADR-0006** — «model глобальна для subagent_type» (поэтому ростер `ds-*` свой, без пина) и «субагенты не
  спавнят субагентов» (поэтому консолидацию находок и единый демо строит оркестратор, аудиторы read-only).
- **ADR-0008 / ADR-0013 / ADR-0016** — агностичный к содержимому бэкенд: находки/демо/гейт поверх
  существующих полей без правок сервера. `/design` — следующий случай того же культа «0 правок»;
  per-finding гейт `f<k>` напрямую реюзит контракт `feat-K` (`questions[choice]` + `approve-plan`) из
  ADR-0013, меняя лишь дефолт (opt-out вместо opt-in).
- **ADR-0020** — вложения-изображения в дашборде (`/attach` → ref → `GET /image`): вход компонента
  скриншотом, оркестратор `Read`'ит сохранённый файл.
- **ADR-0012 / ADR-0014** — рой и очередь `/improve`. `/design` **расширяет** идею роя (несколько read-only
  аудиторов по призмам + консолидация оркестратором), но **без** голосующей панели/детерминированной
  агрегации (ADR-0012) и **без** DISPATCH-очереди `/feature` (ADR-0014) — `/design` реализует сам через
  `ds-coder`, в скоупе одного компонента.
