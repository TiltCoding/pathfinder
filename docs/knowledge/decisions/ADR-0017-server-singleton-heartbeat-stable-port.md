# ADR-0017: надёжность companion-сервера — идемпотентный singleton + heartbeat + стабильный порт

- **Дата:** 2026-06-18
- **Статус:** принято
- **Задача:** server-reliability

## Контекст

Дашборд-сервер (`scripts/server.py`) запускается из скиллов в фоне (`run_in_background`, по разу на
сессию). На практике у пользователя копился **завал осиротевших серверов**: на машине одновременно
жили 5+ процессов `server.py` (`ppid 1`, по 3–6 дней) на портах 8473–8480, а `.workflow/server.json`
указывал на **мёртвый** pid/порт (труп). Симптом «сервер постоянно падает» на деле был обратным —
серверы **не умирали**, а накапливались, и `server.json` терял, который из них канонический; URL из
файла вёл на труп, пока рядом висели живые, но «бесхозные».

Корни:

- `main()` **всегда** биндил новый порт, перезаписывал `server.json` (last-writer-wins) и уходил в
  `serve_forever` — **никогда** не переиспользуя живой сервер и никого не убирая.
- Контракт reuse в скиллах сверял `/health` с `pid`/`port` из `server.json`; раз там был дохлый pid,
  проверка всегда проваливалась → каждая сессия плодила **ещё один** сервер.
- `server.json` не чинился при смерти процесса; `ts` писался один раз на старте (бесполезен как
  «свежесть»); порт «плавал» сканом от запуска к запуску, поэтому разные проекты дрались за 8473.

## Решение

Серверу — **самодедупликация и самоуборка**, а не «демонизировать сильнее» (он и так переживал сессию):

1. **Идемпотентный запуск (singleton).** `main()` читает прежний `server.json`; если он описывает
   **живой** сервер для этого корня (`server_is_live`), печатает `reusing live server …` и **выходит**,
   не поднимая второй. Это корневой фикс: повторный запуск больше не плодит дубликаты. `--force` —
   обойти.
2. **Heartbeat.** Демон-тред `Heartbeat` раз в `HEARTBEAT_SECS` (5 с) переписывает `server.json.ts`.
   Читатель отличает труп с **переиспользованным** ОС-pid по «застывшему» `ts` старше
   `SERVER_STALE_SECS` (30 с) — `server_info_age` → `server_is_live`. Поле `pid`-liveness остаётся
   первичным; heartbeat-свежесть — дополнительный признак (отсутствие/мусор в `ts` → fallback к pid,
   консервативно).
3. **Стабильный порт по корню.** `port_for_root` = `8473 + sha1(realpath(root))[0] % 25` (скан как
   fallback). URL проекта не «плавает», два разных проекта реже целятся в один порт. `hash()` не
   годится — он рандомизирован на процесс.
4. **Уборка осиротевших.** `discover_servers` зондирует диапазон портов через `/health` (теперь он
   отдаёт `root`); чистый `gc_targets` выбирает, кого прибить (фильтр по `realpath(root)`, всегда
   исключая себя); `reap_servers` шлёт `SIGTERM`. Доступно как `server.py --gc` и вызывается перед
   bind при reclaim.
5. **Очистка на выходе.** `install_shutdown_cleanup` (atexit + хендлеры SIGTERM/SIGINT → `sys.exit`)
   удаляет **свой** `server.json` (`clear_server_info` — только если pid в файле наш, чтобы не затереть
   указатель преемника). Так штатная остановка не оставляет трупа.

`server.json` и `/health` получили поле `root` (идентификация проекта). Контракт reuse в
`skills/*/feedback-loop.md` упрощён до «просто запускай — сервер сам дедуплицируется».

## Альтернативы

- **`setsid`/`nohup`/launchd (демонизация).** Отвергнуто как основа: серверы и так переживали сессию
  (`ppid 1`) — проблема не в смерти, а в **накоплении** и потере канонического. Демонизация лишь
  усугубила бы завал.
- **Реестр `runs.json` живых серверов.** Лишнее состояние; `/health`-зонд по детерминированному
  диапазону портов проще и не рассинхронизируется.
- **Случайный/сканируемый порт (как было).** Нестабильный URL + межпроектные коллизии за 8473.

## Последствия

- Один сервер на корень; повторные запуски — no-op-reuse; URL стабилен; трупы не копятся.
- **Переходное:** legacy-серверы старого кода не отдают `root` в `/health` → авто/`--gc`-reap их не
  трогает (консервативно «чужой»); разовый завал убирается вручную (`kill` + `rm server.json`).
- Тесты — `tests/test_server_health.py` (pure-хелперы: `server_info_age`, `server_is_live`,
  `port_for_root`, `gc_targets`; `/health.root`; `server.json.root`). I/O (heartbeat-тред, зонд, kill)
  не юнит-тестируется — проверено живым smoke-прогоном.
- Живой плагин исполняет **кэш-копию** кода (см. `memory`), поэтому фикс вступает в силу после
  обновления плагина из marketplace (bump `0.16.0 → 0.17.0`); для самого репо — запускать
  `scripts/server.py` напрямую. Связь: ADR-0010 (общий store/хаб), архитектура §«Startup-контракт».
