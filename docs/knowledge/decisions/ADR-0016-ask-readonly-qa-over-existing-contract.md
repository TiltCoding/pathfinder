# ADR-0016: `/ask` — read-only Q&A поверх существующего контракта (бейдж `kind` — единственная правка)

- **Дата:** 2026-06-15
- **Статус:** принято
- **Задача:** ask-command

## Контекст

Нужна четвёртая slash-команда плагина — **`/ask`**: лёгкий **read-only** режим «вопрос-ответ». По вопросу
человека оркестратор обследует доки/код, синтезирует **текстовый ответ + инфографику + схему процесса
решения**, показывает их на дашборде и держит **чат** для дальнейших вопросов. Каждый запуск виден в
**хабе** как отдельная «ask»-задача. `/ask` **ничего не правит** в коде проекта.

Разведка (`exploration.md`, `exploration-runtime.md`) показала, что **ни один серверный эндпоинт не
завязан на семантику доменных полей** — модель дашборда чисто клиентская, сервер отдаёт JSON-байты
(подтверждено ADR-0008, ADR-0013). Значит почти весь MVP `/ask` ложится на существующие контракты:

| Возможность `/ask`        | Готовый механизм                                                         | Правки |
|---------------------------|--------------------------------------------------------------------------|:------:|
| Текстовый ответ           | `dashboard.json.summary` (+ опц. `planBlocks`) → GFM-рендер `md()`        |   0    |
| Инфографика               | `demo.variants[]` → самодостаточный HTML в `mockups/` → iframe `/mockup`  |   0    |
| Схема процесса решения    | второй `demo.variants[]` → `process.svg` в `mockups/` (`image/svg+xml`)   |   0    |
| Чат «дальнейшие вопросы»  | `chat.jsonl` + сигнал `chat` + long-poll `/wait`                         |   0    |
| Запись в хаб как задача    | каталог `tasks/<slug>/` + `state.json`(phase,updatedAt) + `dashboard.json`(title) | 0 |

Бриф **прямо требует** «отличать ask от feature/improve» — это единственное требование, которого нет в
существующем контракте: карточка хаба (`_hub_run`/`runCard`/`histRow`) **не знает про тип задачи**.

## Решение

**Строить `/ask` целиком поверх существующего контракта**, с **единственной** осознанной правкой кода —
опциональным бейджем `kind:"ask"` в хабе.

1. **Текст = `summary`/`planBlocks`.** Оркестратор (у него Write) сам пишет markdown-ответ в `summary` и,
   при разбивке, карточки `planBlocks[]` — 0 правок сервера/HTML.
2. **Визуализации = `demo`/`mockups`.** Две самодостаточные визуализации подаются штатным `demo`-блоком
   (`kind:"diagram"`, `variants:[{process.svg}, {infographic.html}]`); сервер отдаёт их через
   `GET /mockup?slug=&file=` в sandbox-iframe. SVG поддержан из коробки (`image/svg+xml`). Имена под маску
   `MOCKUP_RE = ^[A-Za-z0-9._-]{1,64}\.(html|svg)$` (латиница/цифры/`._-`), файлы без CDN. **0 правок.**
3. **Чат = `chat.jsonl`.** Полностью агностичный к типу задачи канал; требование к `/ask` — слушать сигнал
   `chat` через `/wait` и дописывать `role:"agent"`. Это поведение скилла, не рантайма. **0 правок.**
4. **Хаб — автоматически.** `_build_hub` сканирует `.workflow/tasks/*`; каталог `ask-command/` уже
   соответствует ⇒ задача уже в хабе. **0 правок** для попадания.
5. **Единственная правка кода — бейдж `kind`** (append-only поле `state.json`, `conventions.md`):
   `_hub_run` прокидывает `state.get("kind")`; `runCard`/`histRow` рисуют бейдж типа, если `r.kind` задан;
   опц. CSS `.badge.kind`. ~4 строки в `scripts/server.py`, **HTML дашборда не трогается** (хаб живёт в
   `HUB_PAGE` внутри server.py). Задачи без `kind` (`/feature`/`/improve`) рендерятся как раньше.

**Мини-рой ресёрчеров по граням.** RESEARCH спавнит **несколько** read-only `ask-researcher` параллельно
(по граням: доки/база знаний, серверный код, фронт/дашборд, тесты), затем оркестратор **консолидирует**
дайджесты в единую выжимку. Это **расширяет** паттерн роя `/improve`, но **без голосования и без очереди**
(нет CONSENSUS-панели/`score`, нет DISPATCH-очереди) — `/ask` только отвечает, не приоритизирует.

**Синтез/рисование — у оркестратора, не у агента.** Инфографику/схему рисует **сам оркестратор** (у него
Write), а не отдельный `ask-synthesizer`. Причина — инвариант ADR-0006 «**субагенты не спавнят
субагентов**»: «исследователь, который потом сам ответит» допустим, но «синтезатор, который при нехватке
данных сам дочитает код» — нет (агент-синтезатор не сможет до-исследовать). Оркестратор медиирует хэндофф
«ресёрч → синтез», поэтому держит всё рисование у себя. `ask-researcher` идёт **без `model:`** (дефолт
сессии, как `wf-*`) — не плодим новый ростер с пиннингом (ADR-0006).

**Жизненный цикл.** Стадии `INTAKE → RESEARCH → SYNTHESIZE → ANSWER → DONE`. `ANSWER` —
**нетерминальная** чат-петля (`phase` остаётся `ANSWER` между вопросами), иначе хаб посчитает задачу
законченной и уведёт в «Историю» на время чата (`_hub_is_active`, `server.py:792`). **Авто-DONE после
~24 ч** тишины в чате (совпадает с окном «активна» хаба) либо по явному запросу человека.

## Почему

- **Культ «0 правок» (ADR-0008/0013).** Бэкенд агностичен к содержимому → текст/демо/чат/хаб ложатся на
  существующие поля без новых эндпоинтов/флагов; риск рассинхрона фронт↔сервер↔агент нулевой. Единственное
  требование вне контракта (различать тип в хабе) закрыто **минимальной** append-only правкой по уже
  описанному рецепту «новое поле карточки хаба» (`areas/parallel-runs-hub.md`), а не новым эндпоинтом.
- **`/improve` — ближайший шаблон оркестратора:** тоже read-only-производитель без VERIFY и без
  гейта-имплемента. Из него берём стадии-таблицу, чат, модель дашборда, state-схему; **выкидываем**
  SELECT GATE, CONSENSUS, DISPATCH-очередь, рой из 7 scout. Так `/ask` — самый лёгкий из четырёх.
- **Синтез у оркестратора честен и обязателен** (инвариант ADR-0006): без Agent-tool у субагентов
  единственный, у кого есть Write и право спавна, — оркестратор; «синтезатор, дочитывающий код сам» был бы
  невозможен. Поэтому рисование держим у оркестратора, а `ask-researcher` остаётся чисто read-only.
- **Нетерминальная `ANSWER`** — прямое следствие критерия `_hub_is_active` (`phase ∉ {DONE,ABORTED}` И
  свежее 24 ч): держим задачу «активной» в хабе ровно на время чата, авто-DONE синхронизирован с окном.

## Последствия

- **Бейдж `kind` — append-only.** Поле опционально; старые задачи без `kind` рендерятся как раньше
  (бэкомпат проверяется тестом `tests/test_hub.py`: `kind:"ask"` → `run["kind"]=="ask"`; без `kind` →
  отсутствует/`None`). Формат телеметрии/state не ломается.
- **Косметика карточки демо.** `demo`-карточка зовётся «Демо решения» и несёт radio выбора варианта —
  для ответа семантически странновато, но **функционально работает**. Переименование/убор radio
  («Ответ/Инфографика/Схема») — **аддитивный** апгрейд `dashboard.html` (1 ветка рендера), отложен (не для
  MVP) — выбор человека на гейте (q2) = «реюз demo as-is».
- **Схема процесса статична.** Рисуется из секций дайджеста (`## Шаги рассуждения`/`## Опорные источники`),
  **не** из живого `telemetry.jsonl`/таймлайна — чтобы артефакт был самодостаточным и воспроизводимым
  (вкладка «Трейсинг» остаётся живым представлением и не дублируется).
- **Мини-рой без консолидирующего LLM.** Консолидацию дайджестов делает **оркестратор** (как агрегацию
  голосов в ADR-0012) — субагенты не спавнят субагентов, отдельного агента-консолидатора нет.
- **`/ask` — четвёртый потребитель** паттерна «поверх существующего контракта» (после `/feature`-фидбэка,
  `/new-product`-гейтов, `/improve`-гейта).

## Альтернативы (отклонено)

- **Вариант A для хаба (0 правок, различать по `slug`/`title`)** — отклонён: бриф прямо требует бейдж
  «ask»; `~4` строки append-only дешевле, чем хрупкая конвенция имён (`exploration-runtime.md §4`, q1=B).
- **Отдельный `ask-synthesizer` с Write** — отклонён: лишний ростер; рисование и так у оркестратора
  (инвариант ADR-0006), а синтезатор не смог бы до-исследовать при нехватке данных (q3=A — оркестратор).
- **Пиннить модель `ask-researcher`** (как `np-*`) — отклонён: дефолта сессии хватает, лишний ростер с
  `model:` не нужен (ADR-0006, q8 = без `model:`).
- **Один ресёрчер на вопрос** — отклонён в пользу мини-роя по граням (q4=B): широкий вопрос дробится на
  доки/сервер/фронт/тесты и собирается консолидацией; узкий — 1–2 агента.
- **Аддитивный режим рендера «Ответ» в `dashboard.html` сразу** — отложен (q5=A): реюз `demo` as-is, чтобы
  стартовать с нулём правок HTML; переименование — готовый аддитивный апгрейд.

## Связи

- **ADR-0006** — инвариант «субагенты не спавнят субагентов» (поэтому синтез/рисование у оркестратора) и
  «model глобальна для subagent_type» (поэтому `ask-researcher` без `model:`, как `wf-*`).
- **ADR-0008 / ADR-0013** — агностичный к содержимому бэкенд: текст/демо/чат/гейты поверх существующих
  полей без правок сервера. `/ask` — следующий случай того же культа «0 правок».
- **ADR-0010** — общий store через симлинк + хаб `/hub` (карточки из `state.json`, критерий active/history
  `_hub_is_active`). `/ask` попадает в хаб автоматически; бейдж `kind` расширяет карточку по рецепту
  `areas/parallel-runs-hub.md`.
- **ADR-0012 / ADR-0014** — рой и очередь `/improve`. `/ask` **расширяет** идею роя (несколько read-only
  агентов + консолидация оркестратором), но **без** голосующей панели/детерминированной агрегации (ADR-0012)
  и **без** DISPATCH-очереди `/feature` (ADR-0014) — `/ask` только отвечает, ничего не диспетчит.
