# ADR-0019: Автономный дренаж очереди `/improve` — отдельный предикат от `--eval`, флаг на уровне очереди, двухуровневый клапан эскалации

- **Дата:** 2026-06-24
- **Статус:** принято
- **Задача:** improve-autonomous-drain

## Контекст

Очередь `/improve` дренажится по одной фиче через `/feature` (ADR-0014), но **каждая** фича всё равно
упирается в единственный человеческий гейт `/feature` — **PLAN GATE** («Утвердить план»). Поэтому
полностью «уйти и вернуться к готовому» нельзя: дренаж замирает на первом же открытом вопросе плана.
Пользователь явно просит **opt-in autopilot-ветку**: агент сам решает открытые вопросы (sensible default
+ записанное обоснование), доходя до DONE без участия человека.

Технически снятие PLAN GATE уже умеет `--eval`/`AIPF_EVAL=1` (headless-бенчмарк). Соблазн —
переиспользовать eval как «режим без гейта». **Ловушка:** eval **связывает** снятие гейта со снятием
review-гейтов в одной точке — `skills/feature/phases.md:101` («In headless/eval mode, skip the gates»).
Автономному пользовательскому режиму нужно ровно противоположное на VERIFY: гейты `/code-review` +
`/security-review` должны **сохраняться** как страховка качества. Значит eval нельзя реюзать как есть —
надо развязать «skip-gate» и «skip-review».

Сервер агностичен к содержимому очереди (`scripts/server.py` `_queue()` — verbatim passthrough очереди
мимо кэша `_hub`, ADR-0008/0013), поэтому новый признак режима — это **данные в очереди**, без правок
бэкенда. Канонический контракт режима — `skills/improve/dispatch-queue.md` (раздел «Autonomous drain
(opt-in)»); остальные skill-файлы ссылаются на него короткими указателями, чтобы не плодить дрейф
дублированных формулировок.

## Решение

### 1. Автономный режим — ОТДЕЛЬНЫЙ предикат от `--eval` (ключевое)

`autonomous` **независим** от `AIPF_EVAL` — это разные предикаты, не один переключатель:

- На **PLAN GATE** автономный прогон ведёт себя **как eval**: не паркуется, авто-резолвит открытые
  вопросы, авто-апрувит план, идёт в IMPLEMENT.
- На **VERIFY** автономный прогон ведёт себя **как обычный прогон**: `/code-review` + `/security-review`
  **выполняются**, high-severity — «fix-or-justify» до DONE, чат-стиринг остаётся включён.

Это и есть развязка `phases.md:101`: формулировка уточнена до «гейты пропускаются **только** в
headless/eval; в автономном режиме review-гейты выполняются как в обычном прогоне». Читатель никогда не
должен спутать два режима — потому ветка автономного режима на PLAN GATE стоит **рядом** с eval-веткой,
а не слита с ней.

### 2. Флаг на уровне очереди — аддитивный boolean `autonomous`

Триггер режима — **top-level `autonomous:true`** в `.workflow/dispatch-queue.json` (рядом с
`mode`/`source`). Отсутствие/`false` ⇒ сегодняшнее ручное поведение (каждый item паркуется на своём
PLAN GATE). `/improve` стампит флаг на DISPATCH по выбору человека на SELECT GATE; `/feature` читает его
в queue-mode шаге 0. **0 правок сервера** — флаг просто переживает verbatim passthrough `/queue.json`
(тест `tests/test_hub.py::QueueEndpointTest` это страхует). В v1 флаг — **на всю очередь**; per-item
автономность отмечена как возможное будущее расширение, не часть v1 (проще, обратносовместимо).

Кроме поля очереди есть CLI-override: явный аргумент **`--auto`/`--autonomous`** (или эквивалентная
просьба на естественном языке) включает тот же режим для одной инвокации `/feature`. Поле очереди —
канонично, CLI-arg перебивает для конкретного вызова.

### 3. Политика авто-резолва + двухуровневый клапан эскалации

Для каждого открытого вопроса агент выбирает **sensible default** — вариант, который в порядке
приоритета лучше всего (a) совпадает с конвенциями/базой знаний (`docs/knowledge/`), (b) наименее
рискован / наиболее обратим, (c) минимальный скоуп, всё ещё закрывающий критерии приёмки брифа. Резолв
проходит через **двухуровневый клапан**:

- **Мягкая эскалация** — обычная неоднозначность, обратимые решения: НЕ блокируем. Берём дефолт, помечаем
  решение `mode:"escalated"`, фиксируем обоснование, поднимаем в чат + DONE-сводку, идём дальше.
- **Жёсткий блок = обязательный человеческий апрув** _(правка человека, 2026-06-24)_: если решение
  **необратимо / рискует потерей данных / деструктивно** — удаление/перезапись данных или файлов,
  деструктивная миграция, `DROP` таблицы/колонки, переписывание git-истории, breaking-change
  публичного/отдаваемого контракта, bulk-операции, **всё, что не откатывается через git** — агент **НЕ**
  делает это автономно: останавливает срез, помечает `mode:"blocked"`, поднимает вопрос человеку
  (`questions[]` + anchored-строка в `chat.jsonl` с `needsAnswer:true`), **паркуется на `/wait` и ждёт
  явного апрува**, продолжает только после ответа. Этот условный гейт срабатывает **даже** в полностью
  «руки-прочь» `/loop /feature`.

Механизм жёсткого блока — **не новый**: это **та же** парковочная машинерия обычного PLAN GATE
(`awaiting-batch` + `questions[]` + `/wait`), включаемая **условно**, только на опасной необратимости.

### 4. След решений без новых артефактов

Каждое авто-решение → `state.json.questions[]` (`answer` + `rationale` + `mode:"auto"|"escalated"|"blocked"`),
зеркало в `replies.json`/`chat.jsonl` (видно на дашборде задачи). Заметные решения дополнительно → ADR +
строка `task-log.md`. **DONE-сводка перечисляет все авто-решения**, а `blocked`-эскалации — отдельной
секцией. Никаких новых каналов/контрактов — переиспользуем существующие дашборд-каналы.

### 5. Точки писателя/драйва

- **Writer (`/improve`):** на SELECT GATE — один дополнительный `questions[kind:"choice"]` со стабильным
  id `drain-mode` («Прогнать выбранные фичи автономно?», опции «Автономно»/«Вручную (как сейчас)»,
  **дефолт «Вручную»** — безопасный, обратносовместимый). По выбору «Автономно» DISPATCH/`consensus.md`
  стампит `autonomous:true` на top-level очереди.
- **Драйв:** авто-продолжение через **`/loop /feature`**; инвариант «свежий контекст на фичу» сохраняется
  (ровно один ре-инвок `/feature` на item — item-ы не цепляются в одной сессии, ADR-0014).

## Почему так, а не иначе

- **Отдельный предикат, а не реюз eval.** eval — это бенчмарк (skip-gate **и** skip-review разом);
  автономный режим — пользовательский (skip-gate, но review-гейты включены, решения логируются). Слить
  их в один флаг значило бы либо потерять review-страховку в автопилоте, либо тащить логирование решений
  в бенчмарк. Развязка в `phases.md:101` — ядро решения: «как eval на PLAN GATE, как обычный прогон на
  VERIFY».
- **Флаг на уровне очереди, а не per-item (в v1).** Минимальный обратносовместимый аддитив; passthrough
  ⇒ 0 правок сервера (lineage ADR-0008/0013). Per-item автономность не дала бы пользы в первой версии и
  усложнила бы writer-сторону — отложено как расширение.
- **Автономность ≠ безрассудность.** Рутинную неоднозначность агент решает сам (мягкая эскалация);
  опасную необратимость пропускает **только** через человеческий гейт. Переиспользование существующей
  парковки PLAN GATE (а не новый механизм) держит реализацию в промпт-контракте и не плодит контрактов.
- **Прозрачность через существующие каналы.** Автономность не должна быть «чёрным ящиком»: все решения
  видны постфактум на дашборде/в чате/в БЗ. Реюз `questions[]`/`replies.json` вместо нового артефакта —
  тот же принцип «поверх существующего контракта», что и ADR-0008/0013.
- **Единый канонический источник режима.** Полное описание — только в `dispatch-queue.md`; прочие файлы
  ссылаются указателями. Иначе формулировки режима разъехались бы между writer'ом и drainer'ом.

## Последствия / границы

- **Обратная совместимость сохранена.** Отсутствие `autonomous` ⇒ парковка на PLAN GATE как раньше;
  `AIPF_EVAL=1` ⇒ гейты по-прежнему пропускаются (eval-ветка не тронута, `phases.md:101` лишь развязан).
  Обычный ручной `/feature` и ручной дренаж не изменены.
- **0 правок серверной логики.** Режим — данные в очереди; `_queue()` отдаёт `autonomous` дозаписью.
  (Опциональный бейдж автопилота на хабе по `data.autonomous` — чисто транспарентная HTML-правка, если
  включён; механика очереди от него не зависит.)
- **Авто-резолв не отменяет качество.** VERIFY-гейты и «fix-or-justify» остаются стеной; high-severity
  по-прежнему не пускает в DONE без фикса/обоснования.
- **Файлы:** канонический контракт — `skills/improve/dispatch-queue.md` (раздел «Autonomous drain
  (opt-in)» + поле `autonomous` в схеме); writer — `skills/improve/{phases,consensus,SKILL}.md`; drainer —
  `skills/feature/{SKILL,phases,feedback-loop}.md` (ядро — развязка `phases.md:101` + ветка PLAN GATE);
  тест passthrough — `tests/test_hub.py`; сервер агностичен (`scripts/server.py`); README; этот ADR.

## Связь с другими решениями

- **ADR-0014** (последовательная очередь + queue-mode `/feature`): автономный режим — надстройка над тем
  же дренажём; инвариант «свежий контекст на фичу» и `/loop /feature` сохранены.
- **ADR-0008 / ADR-0013** (поверх существующего контракта, бэкенд агностичен к содержимому): `autonomous`
  — ещё одно аддитивное поле, переживающее passthrough без правок сервера; след решений — на существующих
  каналах `questions[]`/`replies.json`.
- **ADR-0007** (контекстный `approve-plan`, автономный build-loop `/new-product`): родственная идея
  «автономный прогон, эскалация к человеку только по делу»; здесь триггер эскалации — не стоп-условия
  цикла, а двухуровневый клапан необратимости. Eval так же режет гейты в `/new-product` — `phases.md:101`
  у `/feature` развязан по тому же мотиву.
