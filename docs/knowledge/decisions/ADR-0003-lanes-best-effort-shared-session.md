# ADR-0003: Лейны (дорожки) — best-effort из-за общего session_id под-агентов

- **Дата:** 2026-06-10
- **Статус:** принято
- **Задача:** realtime-agent-tracing

## Контекст

Бриф требует группировать действия в ленте «по агентам/лейнам». Естественно приписывать каждый `tool.*`
тому под-агенту, что его выполнил. Но (нетривиальный факт из exploration §5): **payload хука Pre/PostToolUse
не содержит идентификатора под-агента** — оркестратор и все его под-агенты делят **один `session_id`**.
Различить исполнителя tool-события из телеметрии в общем случае невозможно. Точный исполнитель известен
только постфактум в транскрипте (`attributionAgent`), но он приходит с задержкой и требует чтения прозы.

## Решение

Лейн определяется **best-effort** на сервере (`_feed_lane`, `scripts/_aipf.py:413`): `build_feed` отслеживает
открытые `subagent.start`/`subagent.end` по сессии; если в момент tool-события открыт **ровно один**
под-агентский спан — действие приписывается ему (`lane = spanId`, `role`); иначе — `lane = "orchestrator"`.

## Почему

- **Это потолок того, что даёт источник.** Точная атрибуция при нескольких параллельных под-агентах
  потребовала бы данных, которых в payload хука нет. Тащить её из транскриптов на горячий путь ленты
  противоречит «лента не читает прозу» (ADR-0001) и даёт задержку.
- **Эвристика «ровно один открытый спан»** покрывает частый случай (последовательные под-агенты или один
  активный) корректно и дёшево, без состояния между запросами.
- **Деградация безопасна.** Неверно приписанное к `orchestrator` действие всё равно отображается с верными
  таймингами; теряется только группировка. Клиент может уточнять по своей накопленной модели.

## Последствия

- При нескольких **одновременных** под-агентах группировка действий по лейнам неточна — это известное и
  принятое ограничение, отражено в подводных камнях области телеметрии.
- При delta-чтении (`since>0`) под-агент, чей `subagent.start` остался до курсора, не виден `build_feed`,
  и его действия деградируют до `orchestrator`; клиент корректирует по своей модели.
- Если позже понадобится точная атрибуция — придётся обогащать ленту данными транскрипта
  (`attributionAgent`) лениво/постфактум, не на горячем пути.
