# ADR-0021: кросс-платформенная атомарная запись — per-process temp + retrying replace

- **Дата:** 2026-06-25
- **Статус:** принято
- **Задача:** atomic-write-pid-temp (дренаж аудита `improve-overall-2`, feat-2 / cand-14)

## Контекст

Параллельные запуски `/feature` делят **один** store через симлинк `<worktree>/.workflow → <main>/.workflow`
(ADR-0010). Значит несколько процессов сервера/хука (а внутри сервера — несколько потоков `HTTPServer`)
могут писать **один и тот же** целевой файл (`state.json`, `draft.json`, `server.json`, `settings.json`)
**конкурентно**.

Прежний `atomic_write` (`scripts/_aipf.py`) делал запись через **общий** temp `path + ".tmp"`, затем
`os.replace`. Эта схема ломается при конкуренции на двух уровнях:

1. **Гонка на temp.** Все писатели целятся в **один** `path.tmp` — их байтовые потоки перемешиваются, и
   победитель `os.replace` может опубликовать **полу-записанный** файл.
2. **Windows-специфика `os.replace`.** На POSIX `os.replace` поверх существующего файла атомарен и
   **никогда** не конфликтует — он успешен с первой попытки. На **Windows** `os.replace` атомарен **по
   результату**, но кидает **транзиентную** `PermissionError(13)` (`ERROR_ACCESS_DENIED`), если в этот
   самый момент destination открыт/переименовывается другим писателем. Без ретрая это всплывает как
   случайный сбой записи под нагрузкой параллельных запусков.

Тот же дефект жил и во втором центральном писателе — `server.Workspace.write_json`, который дублировал
логику temp+replace независимо (риск дрейфа двух реализаций).

## Решение

Кросс-платформенная атомарная запись с конкурентными писателями требует **двух** составляющих —
обе обязательны:

1. **Per-process temp** убирает гонку на temp-файл. `atomic_temp_name(path)` (`scripts/_aipf.py:65`) =
   `path.<pid>.<uuid8>.tmp` — каждый писатель пишет в **свой** temp в том же каталоге, поэтому
   `os.replace` остаётся атомарным (та же FS), но потоки больше не смешиваются.
2. **Ограниченный ретрай replace на транзиентную `PermissionError`.** `atomic_replace(tmp, path)`
   (`scripts/_aipf.py:76`) повторяет `os.replace` до **50 раз с паузой 2 мс** (~100 мс бюджет) **только**
   на `PermissionError`; последняя попытка **re-raise**, так что настоящий сбой (нет прав, нет места) всё
   равно всплывает, а не маскируется. На POSIX ветка ретрая не срабатывает (первая попытка успешна).

`atomic_write` (`scripts/_aipf.py:95`) при любой `OSError` **чистит осиротевший temp** и **re-raise**'ит
исходную ошибку (без тихого фолбэка — повреждённую запись лучше провалить громко). `write_json` чинится
транзитивно через `atomic_write`.

**Один путь записи на оба центральных писателя.** `server.Workspace.write_json` (`scripts/server.py:141`)
переиспользует те же `_aipf.atomic_temp_name`/`atomic_replace` (убран лишний `import uuid`), поэтому две
реализации не дрейфуют. Комментарии в обоих местах ссылаются друг на друга.

Это **переиспользуемый инвариант** для всех будущих писателей в проекте: любой новый атомарный писатель
поверх общего store обязан брать `_aipf.atomic_temp_name` + `atomic_replace`, а не катать свой temp+replace.

## Альтернативы

- **Файловый лок (`msvcrt.locking`/`fcntl.flock`) вокруг записи.** Платформенно-разный, не stdlib-единый,
  и решает только взаимоисключение — не атомарность публикации. Per-process temp + retrying replace
  достигает того же без блокировок и портабельно.
- **Просто ретраить общий `path.tmp`.** Не закрывает гонку на сам temp (полу-записанный файл): два
  писателя всё равно топчут один temp до replace. Per-process temp обязателен.
- **Оставить `os.replace` без ретрая (как на POSIX).** На Windows даёт случайные `PermissionError` под
  конкуренцией — ровно тот баг, который чинили.
- **Глобальный бесконечный ретрай.** Маскирует настоящие сбои (нет прав/места). Бюджет ограничен (50×2 мс)
  и re-raise на исчерпании.

## Последствия

- Конкурентная запись N процессов/потоков в один путь больше не публикует мусор и не падает на Windows.
  Покрыто `tests/test_atomic_write.py` (10 тестов: round-trip обоих писателей, отсутствие осиротевших
  temp, очистка+re-raise на ошибке записи, конкурентная запись N процессов/потоков в один путь).
- Каталог цели на пике конкуренции содержит несколько `*.<pid>.<uuid8>.tmp` — это норма; штатный путь их
  удаляет (через `os.replace`), осиротевшие остаются только при жёстком крахе процесса.
- **Известный долг (НЕ в этой задаче):** два писателя ещё **не** переведены на общий хелпер и используют
  фиксированный/pid-only temp без ретрая — `server.write_lang` (`scripts/server.py:209`, `path + ".%d.tmp"
  % pid`, голый `os.replace`) и attachment-upload (`scripts/server.py:1372`, `path + ".tmp"`). Комментарий
  ~`scripts/server.py:145` ссылается на `write_lang` как на образец «разделяющего приём» — теперь
  неточно. Подтянуть отдельной задачей. Плюс — sweep старых осиротевших `*.tmp`.
- Связь: ADR-0010 (общий store через симлинк — корневая причина конкуренции), `conventions.md`
  §«Полезные утилиты» / §«Технологические ограничения» (инвариант атомарной записи).
