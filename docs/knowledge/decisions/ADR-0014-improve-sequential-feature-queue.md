# ADR-0014: `/improve` DISPATCH — последовательная очередь `/feature` вместо параллельного фан-аута worktree

- **Статус:** принято (2026-06-13, v0.13.0)
- **Контекст:** заменяет модель DISPATCH из ADR-0013 (часть про «как запускаются выбранные фичи»;
  контракт выбора `feat-K` на гейте — без изменений).

## Проблема
DISPATCH `/improve` раньше **сеял каждую выбранную фичу как отдельный `/feature`-прогон в своём git
worktree** и отдавал человеку 8 команд `cd <worktree> && /feature`. На практике (прогон
`improve-runtime`, 8 пиков) это дало два провала:
1. **Запустить N независимых сессий вручную невозможно/мучительно** — ровно та трудозатрата, которую
   воркфлоу должен убирать.
2. Альтернатива «одна сессия делает все N фич подряд» **раздувает контекст** до деградации качества.

## Решение
DISPATCH теперь **пишет очередь и дренажит её по одной фиче через `/feature`, каждую в свежем
контексте**:

- **Очередь:** один проектный файл `.workflow/dispatch-queue.json` (не per-task, чтобы `/feature` нашёл
  его без знания slug аудита) — единственный durable-источник, переживающий `/clear`. Поля:
  `version/source/mode/baseCommit/items[]`; `items[]` в ранговом порядке (`feat-1`→`n:1`) со статусом
  `pending → in-progress → done`. Полный контракт — `skills/improve/dispatch-queue.md`.
- **Писатель (`/improve` DISPATCH):** на каждую фичу пишет только `brief.md` и добавляет `pending`-item.
  **Не** создаёт worktree, **не** сеет per-feature `state.json`/`dashboard.json`/`index.html`, **не**
  запускает `/feature` сам (это бы засорило его контекст). В `state.json.dispatched[]` — облегчённый
  mirror `{slug, featId, candId, briefPath, status:"queued"}` (без `worktreePath`).
- **Дренажер (`/feature` queue-mode):** при запуске без явной задачи и наличии `pending`-item берёт
  младший `n`, помечает `in-progress`, берёт `briefPath` как **готовый бриф** (пропуск INTAKE-элиситации),
  ветвится от `baseCommit` очереди, гонит полный workflow (EXPLORE→…→DONE с гейтом плана и ревью-гейтами),
  на DONE помечает item `done` и просит человека `/clear` + `/feature` для следующего.
- **Две опции драйва (человеку):** «чистый контекст» = `/feature` → `/clear` → `/feature` (рекомендуется);
  «руки прочь» = `/loop /feature` (харнесс сжимает контекст между итерациями).

## Почему так, а не иначе
- **Модель не может сама себе сделать `/clear` посреди хода** → настоящая очистка контекста между фичами
  достижима только ре-инвоком `/feature` в новой сессии. Поэтому драйв — человек/`/loop`, а очередь-файл
  несёт состояние через очистку.
- **`/improve` не запускает `/feature` сам** именно чтобы не тащить работу фичи в свой контекст —
  иначе теряется весь смысл «свежий контекст на фичу».
- **Полное качество `/feature`** (исследование, гейт плана, code/security-review гейты, рост базы знаний)
  сохраняется на каждой фиче — в отличие от лёгкого «один под-агент инлайнит фичу».

## Последствия / границы
- Параллельный фан-аут в worktree (`parallel.md`, ADR-0010) **остаётся как opt-in** — человек может явно
  попросить; это больше не дефолт DISPATCH.
- Хаб `/hub` показывает каждый дренажный прогон по мере выполнения (он store-driven по
  `.workflow/tasks/<slug>/`), как и раньше — просто прогоны идут последовательно, а не разом.
- `feat-K`-контракт выбора, агрегация (ADR-0012) и батч-фидбек гейта — без изменений.
- Файлы: `skills/improve/{SKILL,phases,consensus,state-schema}.md`, новый
  `skills/improve/dispatch-queue.md`, `skills/feature/{SKILL,phases}.md`, `plugin.json` (0.12.0→0.13.0).
