# ADR-0007: Эволюционный build-loop `/new-product` (гибридный гейт, вердикт-объект, заморозка тестов)

- **Дата:** 2026-06-12
- **Статус:** принято
- **Задача:** new-product-workflow

## Контекст

В `/new-product` нет готовой кодовой базы и нет человека, который ревьюит каждую итерацию — продукт
**растится с нуля**, фаза за фазой (вертикальными срезами внутри стадии BUILD). Нужен автономный цикл
«сгенерировать → проверить → улучшить», который (а) не «гоняет по кругу» бесконечно, (б) не даёт модели
**гейминг** (подогнать тест/самооценку под зелёный результат), (в) останавливается и зовёт человека
только по делу. LLM-судья как единственный критерий ненадёжен (дрейф оценки, галлюцинации, поощрение
«зелёных тестов» вместо реального соответствия требованиям). Чистый pass/fail-судья не даёт сигнала для
улучшения. Машина (детерминированная логика) и человек (гейты) должны быть разнесены от вероятностной
части (генерация и суждение). Ядро описано в `skills/new-product/loop.md`; решения гейта зафиксированы
человеком (`.workflow/tasks/new-product-workflow/plan.md:41`, политика **V1**).

## Решение

Цикл на фазу `k` (полный псевдокод — `skills/new-product/loop.md`):

1. **Гибридный гейт: тесты — стена, судья — руль.** Сначала `np-coder` в режиме **tests-first**
   материализует исполняемые тесты из **спеки мыслителя** (не видя плана реализации); оркестратор
   замораживает их. Дальше итерации: код → прогон тестов. **Красные тесты ⇒ судья не вызывается**
   (фидбэк = вывод тестов); фаза **никогда** не закрывается при красных тестах. Судья включается только
   на зелёных и оценивает **соответствие FR**, а не «тесты зелёные».
2. **Вердикт-объект вместо pass/fail.** Судья возвращает структуру по
   `templates/artifacts/judge-verdict.md`: на критерий — `score 0–3` + **evidence** (файл:строка/вывод
   теста) + **fix-инструкция**; плюс `blocking_issues`, `unknowns`, `weighted_total 0–100`. Без evidence
   балл не ставится; готовый код судья предлагать не вправе — только локализация + инструкция.
3. **Изоляция измерений: 1 вызов судьи = 1 измерение рубрики.** Рубрика = **3 измерения**, шкала
   **0–3** на критерий, `PASS_THRESHOLD = 80/100`, ни одного критерия в 0 (`plan.md:44`). Три
   `np-judge` спавнятся **параллельно**, оркестратор мёрджит вердикты (взвешенный total, объединение
   `blocking_issues`; критерии с `Unknown` в плюс не идут).
4. **Замороженные PRD-производные тесты (анти-гейминг).** Спеку тестов пишет **мыслитель**, материализует
   **кодер**, реализует фичу **другой проход кодера**, не видя плана реализации. После материализации
   пути + хэши пишутся в `state.build.phases[k].frozenTests`; на каждой итерации оркестратор сверяет
   freeze (`git diff` по путям); нарушение → revert + эскалация. Кодер не вправе «чинить» тест —
   несогласие со спекой возвращает оркестратору.
5. **Reflexion-scratchpad как память между итерациями.** На REFINE оркестратор пишет дистиллят
   `{iteration, what_failed, hypothesis_why, what_to_change}` (из вывода тестов + fix-инструкций судьи);
   **только дистиллят, никаких сырых логов/диффов**. Следующая итерация кодера стартует со **свежим
   контекстом** + этим scratchpad.
6. **`decision()` считает оркестратор детерминированно** (не LLM): `красные тесты → REFINE;
   blocking_issues → REFINE; зелёные ∧ total ≥ 80 ∧ нет критериев в 0 → PASS; iteration ≥ budget →
   STOP_BUDGET; плато → STOP_PLATEAU; осцилляция → ESCALATE; иначе REFINE`.
7. **Три стоп-условия + анти-осцилляция.** Бюджет ≤5 итераций/фазу; **плато** = прирост лучшего
   `weighted_total` <3 баллов за 2 итерации подряд (с 3-й); **осцилляция** = 3 вердикта подряд без
   нового максимума со сменой знака дельты → **эскалация**. Любое STOP_*/ESCALATE → парковка
   `awaiting-batch` + choice-вопрос человеку («+N итераций» / «re-scope фазы» / «принять как есть» /
   «прервать») с приложенной лучшей попыткой, scratchpad и историей score.
8. **Контекстный `approve-plan`.** Гейтов два (PRD-GATE, PLAN-GATE), сигнал кнопки **один** —
   существующий `approve-plan`; оркестратор интерпретирует его **по текущей стадии**. Никаких правок
   сервера/HTML.
9. **Гейт-политика V1.** Два гейта (PRD + план фаз), build-loop **автономен**: переходы между фазами
   без гейта, эскалации — только по стоп-условиям (`plan.md:49`, `plan.md:277`). На SHIP — холистический
   прогон судьи «продукт vs PRD» (FR-трейсинг). В `AIPF_EVAL=1` — жёсткий cap 2 итерации, эскалации
   авто-резолвятся в «принять лучшее», гейты авто-проходятся.

## Почему

- **Гибридный гейт разделяет «истину» и «руль».** Тесты — детерминированный, неподкупный сигнал
  корректности (стена: красные = не закрываем). Судья даёт **направление** улучшения там, где тесты
  молчат (соответствие FR, качество). Звать судью на красных тестах бессмысленно (и дорого) — сначала
  должно компилироваться и проходить.
- **Вердикт-объект > pass/fail.** Цикл рефайнмента нуждается в **actionable**-сигнале: где именно
  (evidence) и что менять (fix). Pass/fail не даёт кодеру за что зацепиться; evidence-дисциплина
  («без пруфа — нет балла») прямо бьёт по галлюцинациям судьи.
- **Изоляция измерений** уменьшает дрейф: один вызов держит в фокусе один критерий и не «усредняет»
  оценку по всей картине; объединение `blocking_issues` гарантирует, что критичное из любого измерения
  не потеряется.
- **Заморозка PRD-производных тестов — главный анти-гейминг.** Если бы реализующий кодер мог править
  тесты, он бы подгонял их под свой код. Разнесение ролей (мыслитель пишет спеку → кодер материализует →
  другой кодер реализует, не видя плана) + хэш-freeze делают тест **независимым** от реализации. Это
  закреплено в роли кодера (`agents/np-coder.md`) и в operating rules (`skills/new-product/SKILL.md:103`).
- **Scratchpad-дистиллят** даёт памяти между итерациями (учимся на провалах — Reflexion), но не раздувает
  контекст сырыми логами; кодер каждый раз стартует «чисто», получая только выводы, а не шум.
- **`decision()` у оркестратора, а не у LLM.** Решение «закрыть/повторить/остановить» — это арифметика
  (числа тестов, total, счётчик итераций, история). Детерминированный код воспроизводим, не галлюцинирует
  и не подвержен дрейфу; LLM оставлен только там, где он незаменим (генерация и суждение).
- **Три стопа + осцилляция** гарантируют **конечность** автономного цикла: бюджет ловит «не сходится»,
  плато — «застряли близко», осцилляция — «качается без прогресса». Без них greenfield-loop мог бы
  крутиться вечно или жечь бюджет вхолостую.
- **Контекстный `approve-plan`** даёт два логических гейта **без правок сервера/HTML** — кнопка зашита в
  HTML, и переиспользовать её по стадии дешевле и надёжнее, чем плодить новые сигналы.

## Последствия

- Состояние фазы обязано нести `frozenTests[]` (пути + хэши), `scratchpad`, `scoreHistory[]`,
  `budget` — это часть `state-schema.md`; resume входит строго **на границе итерации**
  (`skills/new-product/SKILL.md:80`).
- Вердикты судьи ложатся в `reviews.json` (`kind: "judge"`, `id: "judge-<фаза>-i<итерация>"`,
  `plan.md:39`), тренд score рендерится в `summary` дашборда, эскалации — в `questions[]`. Новых
  серверных эндпоинтов не требуется — всё ложится на существующие render-модели (см.
  `skills/new-product/dashboard-guide.md`).
- Появилась нагрузка на согласованность: имена/пороги/хэш пустого дерева/дефолты цикла должны совпадать
  в `SKILL.md`, `phases.md`, `loop.md`, `state-schema.md`, шаблонах и здесь — при правке **править
  всюду** (канонические константы — `plan.md:19`).
- Каждая фаза стоит дороже простого one-shot: tests-first проход + до 5 итераций × (кодер + 3 судьи).
  Цена осознанная — это плата за анти-гейминг и автономность; `AIPF_EVAL=1` режет её до cap 2 итераций
  для бенчмарков.
- Качество всего цикла держится на **машиночитаемости PRD** (FR-ID → GWT → тест → строка рубрики):
  если мыслитель не выдаст её, трейсинг и судейство рассыпаются — поэтому формат зашит в шаблоны
  (`templates/artifacts/prd.md`, `templates/artifacts/judge-verdict.md`,
  `templates/artifacts/iteration-scratchpad.md`), а не оставлен на вкус модели.
