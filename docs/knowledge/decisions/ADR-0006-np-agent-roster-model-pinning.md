# ADR-0006: Отдельный ростер `np-*` с пиннингом модели через frontmatter

- **Дата:** 2026-06-12
- **Статус:** принято
- **Задача:** new-product-workflow

## Контекст

Команде `/new-product` нужно гонять разные роли на разных моделях: «мыслитель» — на маленькой быстрой
**fable** (он только рассуждает и пишет, не читает сырьё), а исследователь/кодер/судья — на **opus**
(тяжёлый ресёрч, код, суждение). В плагине под-агенты тематически нейтральны, и был соблазн
**переиспользовать существующий ростер `wf-*`** (`wf-coder`, `wf-reviewer` и т.п.) вместо заведения
нового набора файлов.

Ключевой факт из разведки (`.workflow/tasks/new-product-workflow/exploration.md:105`): в плагине **нет
прецедента пиннинга модели** — ни в одном `agents/*.md` поля `model:` не было; оркестратор спавнит
агента, передавая лишь `subagent_type`. Развилки: (а) переиспользовать `wf-*` и переопределять модель
на вызове Agent tool, либо (б) объявить модель декларативно во frontmatter — что требует **отдельных**
файлов под каждую роль с нужной моделью.

## Решение

Завести **отдельный ростер `np-*`** под `/new-product`, по одному файлу на роль, с моделью, закреплённой
полем `model:` (алиас) во frontmatter:

- `agents/np-thinker.md:4` → `model: fable`; `tools: Read, Write, Edit` (`agents/np-thinker.md:5`).
- `agents/np-researcher.md:4` → `model: opus`; `tools: Read, Grep, Glob, Bash, WebSearch, WebFetch`.
- `agents/np-coder.md` → `model: opus`; `tools: Read, Write, Edit, Bash, Grep, Glob`.
- `agents/np-judge.md` → `model: opus`; `tools: Read, Grep, Glob, Bash` (read-only по духу).

`wf-reviewer`/`wf-documenter` **переиспользуются как есть** (их модель — дефолт сессии — устраивает, и
менять её для `/feature` не нужно). Оркестратор выбирает только `subagent_type` — модель он не задаёт
(`skills/new-product/SKILL.md:57`).

## Почему

- **`model` глобален для `subagent_type`.** Модель принадлежит файлу агента и едина для всех его
  вызовов — переопределить per-вызов нельзя. Значит реюз `wf-coder` с другой моделью невозможен: правка
  `wf-*` поменяла бы модель и для `/feature` (`exploration.md:151`). Раздельные ростеры — прямое
  следствие этого ограничения, а не вкусовое разделение.
- **`name` уникален в плагине** — нельзя завести второй агент с `name: wf-coder`; новые роли обязаны
  иметь уникальные имена. Префикс `np-` отделяет greenfield-ростер от feature-ростера `wf-*`.
- **Декларативный пиннинг надёжнее override на вызове.** Вариант (а) — override модели при спавне — в
  репо **не используется** и не подтверждён схемой (Context7 в разведке был недоступен,
  `exploration.md:113`); полагаться на него как на единственный механизм рискованно. Frontmatter —
  наблюдаемый в коде, переживает рефакторинг промптов и логически верен: модель принадлежит роли, а не
  конкретному вызову.
- **fable для мыслителя — осознанная экономия.** Мыслитель никогда не читает сырьё (только курированные
  дайджесты), поэтому ему не нужен большой контекст; держать его на тяжёлой opus — расточительно. Малую
  модель «дисциплинирует» сам пайплайн (вход — только дистиллят).
- **Инструменты как структурная гарантия роли.** У `np-thinker` намеренно нет Grep/Glob/Bash/Web
  (`agents/np-thinker.md:5`) — это **структурно** (а не только текстом промпта) запрещает ему читать
  сырьё: «distilled inputs in, contracts out» (`agents/np-thinker.md:14`). Урезанный tool-set —
  машинно-проверяемая часть инварианта «мыслитель видит только выжимки», и он же делает безопасным
  выбор маленькой модели. Симметрично `np-judge` оставлен read-only (без Write/Edit) — судья не правит
  код, только смотрит и выносит вердикт.

## Последствия

- В плагине теперь **два параллельных ростера**: `wf-*` (feature, без `model:`) и `np-*` (new-product,
  с `model:`). При правках помнить: общие роли `wf-reviewer`/`wf-documenter` обслуживают **обе** команды
  — менять их инструкции/инструменты осторожно.
- Появился **первый прецедент** поля `model:` в `agents/*.md`. Пост-фактум наблюдаемость (вычитка
  `model` из транскриптов для трейсинга/Langfuse) — это другое, конфигурацией запуска не была и не
  конфликтует.
- Значение алиаса модели (`fable`/`opus`) проверяется **на спавне**: неверная строка тихо не сработает
  или упадёт — алиасы нужно держать актуальными.
- Если позже понадобится та же роль на третьей модели — это снова **новый файл** агента, не override.
- Урезанный tool-set мыслителя означает: любой факт, которого нет в его курированных входах, он обязан
  пометить как Assumption или вопрос оркестратору (который сходит к исследователю/человеку) — мыслитель
  **не может** добрать данные сам. Это заложено в роль (`agents/np-thinker.md:14`).
