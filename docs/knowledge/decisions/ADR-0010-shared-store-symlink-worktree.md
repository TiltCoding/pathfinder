# ADR-0010: Общий store через симлинк `<worktree>/.workflow → <main>/.workflow` + `worktreePath` в state

- **Дата:** 2026-06-13
- **Статус:** принято
- **Задача:** parallel-runs-hub

## Контекст

Нужно гонять несколько `/feature`-задач **параллельно**, чтобы их рабочие файлы и ветки не дрались.
Естественное решение — git worktree на задачу (своя ветка, своё рабочее дерево). Но один компаньон-
сервер один на проект и читает **один** общий store `<main>/.workflow/tasks/`, а каталог `.workflow/`
**gitignored** — значит в каждом worktree он был бы **локальным и пустым**, и хаб не увидел бы чужие
задачи: артефакты разъехались бы по worktree-каталогам.

Развилки:
- **Как свести артефакты worktree в общий store** хаба: (а) симлинк `<worktree>/.workflow → <main>/.workflow`;
  (б) переменная окружения `AIPF_WORKFLOW_ROOT`, указывающая хук/сервер на main-store; (в) реестр
  `runs.json` в main-store, куда каждый worktree регистрирует свои пути.
- **Как сервер узнаёт рабочее дерево задачи** для вкладки «Изменения» (diff надо считать в worktree,
  иначе он пуст/неверен).
- **Атрибуция сессий** при общем store (одна `active.json` на всех).
- **Где показывать токены** в кросс-задачном агрегате.
- **Хаб — отдельная страница `/hub` или вкладка** внутри дашборда задачи.

## Решение

1. **Симлинк `<worktree>/.workflow → <main>/.workflow`** создаёт `scripts/worktree.py`
   (`_ensure_workflow_symlink`). Артефакты всех задач текут в один store, который читает единственный
   сервер. **Ноль структурных правок серверу** для агрегации (он и так обходит `_list_tasks()`).
2. **Per-task `worktreePath`/`branch` в `state.json`** (пишет `worktree.py`, читает сервер). Сервер
   диффит вкладку «Изменения» в каталоге задачи: `_task_root(slug)` → `git -C worktreePath`, с
   fallback на main при отсутствии/невалидности поля (`_git(*args, cwd=…)`).
3. **Per-session `active/<session_id>.json`** дополнительно к общей `active.json` — чтобы конкурентные
   сессии не перезатирали атрибуцию сессионных событий (`active_slug` предпочитает per-session файл).
4. **Кросс-задачная аналитика — событийная, без токенов/cost.** Хаб строится одним проходом
   `telemetry.jsonl` (`_hub_telemetry`/`_hub_analytics`), `build_trace`/транскрипты не зовутся.
5. **Хаб — отдельная страница `/hub`** (`HUB_PAGE`, инлайн-HTML), развитие лендинга, а не вкладка
   дашборда задачи.

## Почему

- **Симлинк vs env vs реестр.** Симлинк — наименее инвазивный: сервер и хук продолжают читать
  `<root>/.workflow` как обычный путь, ни одна строка резолва пути не меняется. **Env
  `AIPF_WORKFLOW_ROOT`** потребовал бы прокинуть переменную **в хук** (его запускает Claude Code, а не
  мы) — это под вопросом и хрупко. **Реестр `runs.json`** добавил бы записывающий путь, конкурентную
  запись и больше правок в нескольких местах. Симлинк даёт тот же результат бесплатно на уровне ФС.
- **`worktreePath` в `state.json`, а не отдельный файл.** `state.json` — единственный носитель
  per-task контекста, который сервер **уже** читает (`_base_commit`). Добавляем поля «только
  дозаписью» — старые задачи совместимы (поля отсутствуют → fallback на main).
- **Per-session `active.json` обязателен.** При общем store единственная `active.json`
  перезаписывается параллельными сессиями, и `SessionStart/Stop` приписываются не той задаче.
  Per-session ключ чинит это без слома одиночного сценария (нет файла → старое поведение).
- **Токены вне кросс-задачного агрегата.** Токены/cost требуют **транскриптов** (`~/.claude/...`),
  которых в worktree **физически нет**, а `build_trace` дорог (читает файл целиком + транскрипты).
  Замер прототипа: все задачи без транскриптов ≈ 6.6 мс. Поэтому хаб — событийный и дешёвый; токены
  открываются лениво на дашборде конкретной задачи (где её транскрипты доступны).
- **Отдельная страница `/hub` vs вкладка.** Хаб **не привязан к slug** (он про все задачи), а весь
  дашборд и его вкладки параметризованы `?slug=`. Класть кросс-задачный обзор внутрь дашборда одной
  задачи семантически неверно; отдельная страница — естественное развитие лендинга, инлайн-строкой в
  `server.py` (как лендинг), без нового механизма отдачи шаблонов.

## Последствия

- **Per-session `active.json` становится частью контракта параллельного запуска** — оркестратор обязан
  писать `active/<session_id>.json`, иначе атрибуция сессионных событий поедет (см.
  `skills/feature/parallel.md`).
- **Токенов в `/hub.json` нет по дизайну** — кросс-задачные числа всегда событийные; за токенами идти
  на `/?slug=<slug>` (там `/trace` с транскриптами).
- **knowledge-граф наследует worktree-корень.** `_build_knowledge` переиспользует `_build_changes`,
  который теперь диффит worktree задачи → touched-пути относительны worktree-дерева, не main (учитывать
  при сопоставлении с узлами main-репо).
- **Сессию нужно запускать внутри worktree** — только тогда `cwd` хука резолвится через симлинк в общий
  store; запуск из main-дерева сломает атрибуцию.
- **Симлинки на Windows** требуют прав/Developer Mode; хелпер ловит `OSError` и не падает, но без
  рабочего симлинка общий store не функционирует — ограничение платформы, не баг.
- **Одиночный `/feature` не затронут:** без `worktreePath`/per-session файла всё работает как раньше
  (fallback на main / на `active.json`).
