# Вердикт-объект судьи

> Контракт между `np-judge` и оркестратором. **1 вызов судьи = 1 измерение рубрики** — судья
> возвращает объект ниже с **одним** элементом в `per_criterion`. Оркестратор собирает 3 таких
> вердикта (по числу измерений) в один итоговый вердикт фазы по правилам в конце файла.
>
> Судья судит **соответствие FR**, а не «тесты зелёные». На каждый балл — **evidence** (файл:строка
> или вывод теста). Нет evidence → ставится `Unknown`, а не балл наугад. Запрещено предлагать готовый
> код — только локализация проблемы + инструкция как чинить.

## Схема (один вызов = одно измерение)

```json
{
  "per_criterion": [
    {
      "id": "C1",
      "name": "correctness",
      "score": 2,
      "weight": 0.5,
      "evidence": "src/timer.py:48 — пауза не останавливает фоновый тик (наблюдаемо в выводе test_timer.py::test_pause)",
      "fix": "В Timer.pause() выставлять self._running=False до возврата; тик читать этот флаг."
    }
  ],
  "weighted_total": 67,
  "verdict": "revise",
  "actionable_critique": [
    "src/timer.py:48 (пауза не глушит тик) → останавливать тик по флагу _running в pause()"
  ],
  "blocking_issues": [],
  "unknowns": []
}
```

## Поля

| Поле | Тип | Значение |
|------|-----|----------|
| `per_criterion[]` | массив | По одному элементу **на измерение** (в одиночном вызове — ровно один). |
| `per_criterion[].id` | строка | Стабильный id критерия (`C1`…), берётся из рубрики фазы. |
| `per_criterion[].name` | строка | Имя измерения (`correctness` / `UX-clarity` / … — из рубрики). |
| `per_criterion[].score` | 0–3 | Балл по шкале 0–3. **0 запрещён как проходной** — любой 0 валит фазу. |
| `per_criterion[].weight` | 0–1 | Вес измерения (из рубрики); сумма весов всех измерений = 1. |
| `per_criterion[].evidence` | строка | Файл:строка или вывод теста, подтверждающий балл. **Без evidence балл не ставится** — тогда `score` опускается, измерение идёт в `unknowns`. |
| `per_criterion[].fix` | строка | Локализация + инструкция как улучшить (ядро Self-Refine). **Не код.** |
| `weighted_total` | 0–100 | `Σ (score/3 · weight) · 100` по критериям этого вызова (оркестратор пересчитает по всем). |
| `verdict` | `pass`\|`revise`\|`block` | `block` — есть blocking_issue; `pass` — критерий силён и не блокирует; `revise` — иначе. |
| `actionable_critique[]` | массив строк | `<локализация> → <инструкция как чинить>`. То, что попадёт в `findings` дашборда. |
| `blocking_issues[]` | массив строк | Дефекты, при которых фаза не может закрыться (нарушение FR, краш, дыра). → finding `severity: "high"` (классы дашборда — `high`/`med`/`low`). |
| `unknowns[]` | массив строк | Где данных не хватило для балла (escape-люк против галлюцинаций). |

## Правила merge нескольких измерений

Оркестратор получает по одному вердикту на каждое измерение и **детерминированно** сводит их в
итоговый вердикт фазы (это код, не LLM):

1. **`weighted_total` фазы** = `Σ over измерений (score/3 · weight) · 100`, округление до целого.
   Критерии в `unknowns` (без балла) **не засчитываются в плюс** — берутся как 0 вклада, но не как
   проходной 0 (см. п.4); их наличие — сигнал «данных не хватило», а не «качество нулевое».
2. **`blocking_issues`** фазы = объединение `blocking_issues` всех измерений. Любой непустой → итоговый
   `verdict = block`.
3. **`verdict` фазы**: `block`, если есть хоть один blocking_issue; иначе `pass`, если
   `weighted_total ≥ PASS_THRESHOLD` (по умолчанию 80) **и** ни одного критерия со `score = 0`; иначе
   `revise`.
4. **Жёсткое условие нуля:** хотя бы один критерий со `score = 0` → фаза **не** PASS, даже при
   `weighted_total ≥ порога` (одно проваленное измерение нельзя «добрать» весами других).
5. **`actionable_critique` / `unknowns`** фазы = конкатенация одноимённых массивов всех измерений
   (порядок — по `id` критерия).
6. Итоговый объект фазы имеет ту же форму, что и одиночный вердикт, но `per_criterion[]` содержит
   запись на каждое измерение. Он же пишется в `reviews.json` (`kind: "judge"`,
   `id: "judge-<фаза>-i<итерация>"`) — формат summary/findings см. `dashboard-guide.md`.

> Замечание: вердикт судьи — **достаточное** условие качества; красные тесты (независимый сигнал) —
> **необходимое** условие. При красных тестах судья не вызывается вовсе, фаза не закрывается никогда —
> см. `decision()` в `loop.md`.
